# Issue-driven Workflow Standard

이 문서는 작업 단위를 GitHub Issue로 관리하기 위한 최소 실행 규칙이다.

## 1) 시작 게이트
- 구현 작업은 Issue가 있어야 시작한다.
- 예외는 오탈자/문서 수정처럼 코드 영향이 없는 작업만 허용한다.

## 2) Issue 필수 항목
- 배경: 왜 필요한지
- 범위: 무엇을 변경하는지
- DoD: 완료 조건
- 검증 계획: 어떤 명령/시나리오로 확인할지
- 의존관계: 선행/후행 이슈 또는 PR

## 3) 브랜치 규칙
- 권장 형식: `task/<issue-number>-<slug>`
- 한 브랜치는 하나의 주된 Issue를 가진다.

## 4) PR 규칙
- PR 본문에 이슈 연결 키워드를 반드시 포함한다.
  - 예: `Closes #123`, `Fixes #123`, `Resolves #123`
- 검증 근거를 포함한다.
  - 실행 명령
  - 결과 요약
  - 수동 검증 항목(해당 시)
- context/memory 반영 결과를 포함한다.
  - 승격 시 불필요한 단기 내용을 제거했는지
  - 무관 주제를 분리했는지
  - memory 파일명이 `memory-<domain>-<topic>.md` 규칙을 따르는지

## 5) 종료 규칙
- 머지 후 이슈 코멘트에 아래를 남긴다.
  - 반영 내용 요약
  - 검증 결과
  - 후속 작업 필요 여부
- 이슈 종료 시 해당 issue context는 기본 검색 대상에서 제외한다.
- 장기 재사용 가치가 있는 내용만 memory로 승격하고 나머지 단기 내용은 폐기한다.
- context 문서 상태는 `archived` 또는 `superseded`로 전환한다.

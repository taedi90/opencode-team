# Verification Loop 표준

변경 전/후 검증을 같은 루프로 묶어, 재시도 시에도 근거가 남도록 하는 운영 가이드입니다.

## 변경 전 체크
- 대상 변경의 범위를 적는다: 파일, 기능, 영향 경로.
- 현재 상태를 캡처한다: 정상 동작/실패 동작, 재현 조건.
- 검증 명령을 정한다: 실행 명령, 입력 데이터, 기대 결과.
- 기준선(Baseline)을 남긴다: 작업 context 문서의 Verification Log에 기록.

## 변경 후 체크
- 변경 전과 동일한 시나리오/명령으로 다시 검증한다.
- 기대 결과와 실제 결과를 비교한다.
- 부작용을 확인한다: 연관 모듈, 회귀 가능 지점.
- 결과를 context 문서의 Verification Log에 추가하고 상태를 갱신한다.

## 공통 체크리스트
- [ ] 검증 대상과 범위를 명시했다.
- [ ] 변경 전 기준선 결과를 기록했다.
- [ ] 변경 후 동일 시나리오를 재실행했다.
- [ ] 성공/실패 근거(로그, 출력, 관찰)를 남겼다.
- [ ] 실패 시 원인과 다음 액션을 남겼다.

## 재시도 루프
1. 실패 원인을 한 줄로 가설화한다.
2. 최소 수정으로 1회 변경한다.
3. 동일 검증 시나리오를 재실행한다.
4. 결과를 로그에 남기고, 실패 시 1단계로 돌아간다.

원칙:
- 한 번에 하나의 가설만 검증한다.
- 검증 시나리오는 루프 동안 바꾸지 않는다.
- 성공 조건이 충족되면 즉시 루프를 종료한다.

## context 기록 예시
```md
## Verification Log
- 2026-02-20 20:10 pre-check: `npm test -- verification` 실패(2건), 대상 `src/verify.ts`
- 2026-02-20 20:18 attempt-1: null guard 추가 -> 동일 테스트 재실행, 실패(1건)
- 2026-02-20 20:25 attempt-2: 경계값 조건 수정 -> 동일 테스트 재실행, 통과
- 2026-02-20 20:27 post-check: 연관 `npm test -- parser` 통과, 회귀 없음
```
